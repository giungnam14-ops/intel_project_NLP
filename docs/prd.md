# PRD: 5초 문서체크

## 1. 제품명

5초 문서체크

## 2. 한 줄 소개

긴 생활 문서를 NLP로 분석해 문서 유형을 분류하고, 사용자가 놓치기 쉬운 핵심 정보와 주의 문장을 카드와 체크리스트로 보여주는 문서 확인 서비스

## 3. 문제 정의

사용자는 약관, 동의서, 공지문, 안내문, 계약서, 논문·보고서처럼 긴 문서를 자주 접한다.

하지만 대부분의 문서는 길고 표현이 어렵기 때문에 처음부터 끝까지 읽기 부담스럽다. 또한 중요한 정보가 문서 중간에 흩어져 있어 사용자가 직접 찾아야 한다.

예를 들어 약관에서는 자동결제, 환불 제한, 개인정보 제공, 해지 조건을 놓치기 쉽고, 공지문에서는 날짜, 장소, 준비물, 제출 기한을 놓치기 쉽다. 논문이나 보고서에서는 연구 목적, 연구 방법, 핵심 결과, 한계점을 빠르게 파악하기 어렵다.

기존 요약 서비스는 문서 내용을 짧게 줄여줄 수는 있지만, 사용자가 실제로 무엇을 확인해야 하는지 문서 유형별로 직관적으로 보여주지는 못한다.

따라서 긴 문서를 다시 읽지 않아도 핵심 정보와 주의 문장을 5초 안에 확인할 수 있는 서비스가 필요하다.

## 4. 대상 사용자

- 긴 생활 문서를 끝까지 읽기 부담스러워하는 일반 사용자
- 약관이나 동의서에서 중요한 조건을 빠르게 확인하고 싶은 사용자
- 공지문이나 안내문에서 날짜, 장소, 준비물, 제출 기한을 놓치기 쉬운 사용자
- 계약서에서 금액, 기간, 해지 조건, 책임 조항을 빠르게 확인하고 싶은 사용자
- 논문이나 보고서에서 핵심 내용만 빠르게 파악하고 싶은 사용자
- 문서를 읽어도 무엇을 확인해야 하는지 바로 정리되지 않는 사용자

## 5. 핵심 페인 포인트

- 문서가 길고 어려워 처음부터 끝까지 읽기 부담스럽다.
- 중요한 정보가 문서 중간에 흩어져 있어 직접 찾아야 한다.
- 요약문을 봐도 실제로 무엇을 확인해야 하는지 바로 알기 어렵다.
- 문서 유형마다 중요한 정보가 다른데 일반 요약은 이를 구분하지 못한다.
- 약관, 공지문, 계약서, 논문 등 문서마다 필요한 정보가 다르다.
- 사용자는 문서를 읽기보다 대충 넘기거나 Ctrl+F로 필요한 단어만 검색한다.
- 중요한 조건을 놓치면 불편이나 손해로 이어질 수 있다.

## 6. 핵심 사용자 시나리오

사용자가 긴 생활 문서를 읽기 부담스러울 때, 5초 문서체크에 문서를 붙여넣는다.

서비스는 입력된 문서를 분석해 문서 유형을 분류한다.  
그 후 문서 유형에 맞는 핵심 정보와 주의 문장을 탐지한다.

분석 결과는 긴 요약문이 아니라 카드와 체크리스트 형태로 제공된다.

사용자는 문서를 끝까지 읽지 않아도 5초 안에 중요한 내용을 확인하고 다음 행동을 결정할 수 있다.

## 7. MVP 성공 기준

사용자가 긴 생활 문서를 입력했을 때, 서비스가 문서 유형을 분류하고 문서별 핵심 정보와 주의 문장을 카드 및 체크리스트 형태로 보여주면 MVP는 성공이다.

- 약관/동의서 분석
- 공지문/안내문 분석
- 논문/보고서 분석
- 핵심 정보 추출
- 주의 문장 탐지
- 쉬운 말 설명 제공

- 모바일 앱 배포

## 10. 핵심 기능 요구사항

### 10.1 문서 입력 기능

#### 설명

사용자가 분석할 문서를 텍스트로 입력할 수 있다.

#### 요구사항

- 사용자는 긴 문서를 텍스트 입력창에 붙여넣을 수 있어야 한다.
- 입력값이 비어 있으면 안내 메시지를 보여준다.
- 예시 문서를 선택해 테스트할 수 있어야 한다.
- 입력한 문서는 서버로 전송되어 분석된다.

### 10.2 문서 유형 분류 기능

#### 설명

입력된 문서가 어떤 유형인지 자동으로 분류한다.

#### 분류 유형

- 약관 / 동의서
- 공지문 / 안내문
- 논문 / 보고서
- 기타

#### 요구사항

- 분석 결과 상단에 문서 유형을 표시한다.
- 문서 유형에 따라 다른 분석 항목을 보여준다.
- 분류가 불확실한 경우 기타 문서로 표시한다.

### 10.3 핵심 정보 추출 기능

#### 설명

문서 유형별로 사용자가 확인해야 할 핵심 정보를 추출한다.

#### 요구사항

- 약관에서는 자동결제, 환불, 개인정보, 해지 조건을 탐지한다.
- 공지문에서는 날짜, 장소, 준비물, 제출 기한을 탐지한다.
- 논문에서는 연구 목적, 방법, 결과, 한계점을 정리한다.
- 탐지된 항목은 카드 형태로 보여준다.

### 10.4 주의 문장 탐지 기능

#### 설명

사용자가 놓치면 불편이나 손해가 생길 수 있는 문장을 탐지한다.

#### 요구사항

- 주의 문장은 카테고리별로 분류한다.
- 각 주의 문장에는 쉬운 설명을 붙인다.
- 필요하면 원문 문장을 함께 보여준다.
- 법률 판단처럼 보이지 않도록 “확인 필요 항목”으로 표현한다.

### 10.5 쉬운 말 설명 기능

#### 설명

어려운 문장을 사용자가 이해하기 쉬운 문장으로 바꿔 설명한다.

#### 예시

원문:

회원은 무료체험 종료 후 별도 고지 없이 월 이용료가 자동 결제되는 것에 동의합니다.

쉬운 설명:

무료 기간이 끝나면 따로 알려주지 않아도 매달 요금이 결제될 수 있습니다.

### 10.6 카드형 결과 출력 기능

#### 설명

분석 결과를 긴 문장 요약이 아니라 카드 형태로 보여준다.

#### 카드 구성

- 카테고리
- 핵심 내용
- 쉬운 설명
- 중요도 또는 위험도
- 원문 문장

### 10.7 체크리스트 생성 기능

#### 설명

사용자가 확인해야 할 항목을 체크리스트로 제공한다.

#### 예시

약관/동의서:

- 자동결제 시작일 확인
- 환불 가능 여부 확인
- 개인정보 제공 여부 확인
- 해지 마감일 확인

공지문/안내문:

- 날짜 확인
- 장소 확인
- 준비물 확인
- 제출 기한 확인

논문/보고서:

- 연구 목적 확인
- 연구 방법 확인
- 핵심 결과 확인
- 한계점 확인

## 12. 사용자 플로우

1. 사용자가 문서를 입력한다.
2. 사용자가 분석하기 버튼을 누른다.
3. 프론트엔드가 입력 텍스트를 FastAPI 서버로 전송한다.
4. FastAPI 서버가 문서 유형을 분류한다.
5. FastAPI 서버가 핵심 정보와 주의 문장을 탐지한다.
6. FastAPI 서버가 분석 결과를 JSON으로 반환한다.
7. 프론트엔드가 결과를 카드와 체크리스트로 표시한다.
8. 사용자는 5초 안에 핵심 내용을 확인한다.
9. 필요한 경우 원문 문장을 확인한다.

## 13. 기술 구현 방향

### 13.1 구현 방식

MVP는 FastAPI 기반 백엔드와 React 또는 Next.js 기반 프론트엔드로 구현한다.

프론트엔드는 사용자가 문서를 입력하고 분석 결과를 카드와 체크리스트 형태로 확인하는 화면을 담당한다.

백엔드는 입력된 문서를 분석해 문서 유형 분류, 핵심 문장 탐지, 쉬운 말 설명 생성, 체크리스트 생성을 담당한다.

### 13.2 시스템 구조

- Frontend: React 또는 Next.js
- Backend: FastAPI
- NLP Logic: Python 기반 문서 분석 로직
- API: REST API 방식
- Response Format: JSON

### 13.3 처리 흐름

1. 사용자가 프론트엔드에서 문서를 입력한다.
2. 프론트엔드는 입력 텍스트를 FastAPI 서버의 분석 API로 전송한다.
3. FastAPI 서버는 문서를 문장 단위로 분리한다.
4. 문서 유형을 분류한다.
5. 문서 유형에 맞는 핵심 정보와 주의 문장을 탐지한다.
6. 쉬운 설명과 체크리스트를 생성한다.
7. JSON 형태로 분석 결과를 프론트엔드에 반환한다.
8. 프론트엔드는 결과를 카드와 체크리스트 UI로 보여준다.

## 15. 중간 데모 피드백 반영 요구사항

> 본 절은 DAY 11 중간 데모 및 내부 점검 결과([feedback.md](./feedback.md))를 바탕으로 도출한 추가 제품 요구사항이다. 외부 사용자 대상 정량 인터뷰는 수행하지 않았으며, 내부 점검 기준으로 정리한 내용이다. 모든 항목은 현재 MVP 범위(규칙 기반 분석 + 경량 RAG-like + MVP Guardrail)와 일관성을 유지한다.

### 15.1 결과 카드 중요도 표시 (F-001)

- 분석 결과 카드는 중요도(`level`: 높음/보통)를 시각적으로 구분해 표시한다.
- 높음 항목은 🔴, 보통 항목은 🟡 아이콘 또는 색상 코드로 표시한다.
- 목적: 사용자가 우선 확인해야 할 항목에 시간을 집중하고, 보통 항목은 빠르게 검토할 수 있도록 한다.
- 반영 영역: Frontend(결과 카드 렌더링). 백엔드 응답의 `level` 필드를 그대로 활용한다.

### 15.2 체크리스트 행동 문장 강화 (F-002)

- 체크리스트 항목은 단순 안내("~을 확인하세요")가 아니라 행동을 유도하는 문장("~이(가) 맞는지 꼭 확인하세요")으로 제공한다.
- 목적: 사용자가 체크리스트를 읽고 바로 다음 행동을 결정할 수 있도록 긴급성과 행동성을 전달한다.
- 반영 영역: 체크리스트 생성 로직(템플릿을 advisory → action-oriented로 개선).

### 15.3 개인정보 마스킹 안내 문구 (F-004)

- 분석 결과 화면 상단에 "🔐 개인정보가 자동으로 마스킹되어 분석되었습니다" 안내 배너를 표시한다.
- 표시 조건: 입력에서 PII가 탐지되어 마스킹이 적용된 경우(`guardrail_applied=true` 및 관련 경고 존재).
- 목적: 사용자가 민감한 정보를 안심하고 입력할 수 있도록 신뢰도를 높인다.
- 반영 영역: Frontend(결과 요약 영역). 백엔드의 `warnings`/마스킹 여부 신호를 활용한다.

### 15.4 Guardrail 차단 메시지 UX (F-005)

- 입력이 차단된 경우 기술 용어(예: `prompt_injection_detected`)를 그대로 노출하지 않고 사용자 친화적 문구로 표시한다.
  - 예: `prompt_injection_detected` → "시스템 지시를 포함하는 입력은 분석할 수 없습니다."
- 목적: 사용자가 차단 사유를 이해하고 입력을 수정해 재시도할 수 있도록 한다.
- 반영 영역: 백엔드(차단 사유 → 사용자 메시지 매핑) + Frontend(차단 메시지 표시).

### 15.5 문서 유형 수동 선택/수정 옵션 (F-003)

- 자동 분류 결과가 부정확할 수 있으므로 사용자가 문서 유형을 직접 선택/수정할 수 있는 옵션을 제공한다.
- 요구사항:
  - `/analyze` 요청에 문서 유형을 지정하는 `document_type` 값을 사용자가 `auto` 외의 값으로 전달할 수 있다(약관/공지/논문/기타).
  - Frontend에 문서 유형 선택 드롭다운을 추가한다.
- 목적: 잘못된 자동 분류로 인한 부정확한 분석 결과를 방지한다.
- 반영 시점: 다음 스프린트(MVP 이후).

### 15.6 도메인 가이드 출처/설명 표시 (F-008)

- 결과 카드에 도메인 가이드(`domain_guide`)와 주의사항(`domain_caution`)을 함께 표시하고, 해당 정보의 출처를 명시한다.
- 표시 항목: 도메인 가이드 설명 문구 + 출처 표시(예: "도메인 가이드" / `domain_knowledge` 기반).
- 목적: 추가 제공 정보의 신뢰성을 사용자에게 전달한다.
- 반영 영역: 응답 스키마(출처 표시 필드) + Frontend(카드 내 출처/설명 렌더링).
